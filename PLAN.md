# Post-Synthesis Quality Fixes — Phase 2

## Context

Deep research session `deepres-b1fd3155df39` revealed several bugs and gaps in the post-synthesis pipeline introduced in the 0.18.0a8–a9 cycle. The session produced a 33K-char report with 37 sources, 50 claims extracted (36 verified), and a fidelity score of 0.44. Two corrections were applied, but additional issues remain.

### Observed Issues

1. **Heading/body text concatenation after correction** — When `_correct_single_claim()` replaces a context window, the LLM's corrected text can merge a heading with the following body paragraph. In the session, two `###` headings had their body text concatenated onto the heading line (e.g., `### Chase Ultimate RewardsAs of February–March 2026...`). Root cause: the LLM is given a context window that includes the heading + body, and returns corrected text without preserving the `\n` between them.

2. **Duplicate sources sections** — The report contains an inline `*Sources: [2] [LendingTree](...), ...*` italic paragraph (line 194 in the output file) generated by the synthesis LLM, PLUS the deterministic `## Sources` section appended by `postprocess_citations()`. The `strip_llm_sources_section()` function only matches `## Sources`/`## References` markdown headings — it doesn't catch inline/italic source lists that the LLM generates as running text.

3. **Low fidelity score (44%) driven by citation misattribution** — 14/36 claims were `UNSUPPORTED` not because the claims were factually wrong, but because they cited sources that don't contain the supporting evidence. The LLM attributed facts to topically-similar but wrong sources. The existing citation-accuracy guardrails in the synthesis prompt (commit `9ebb084`) didn't fully prevent this.

4. **No corrections for UNSUPPORTED/PARTIALLY_SUPPORTED claims** — The correction pipeline (`apply_corrections()` in `workflow_execution.py:474`) only fires for `claims_contradicted > 0`. Claims with wrong citations but correct facts are left uncorrected.

## Plan

### Fix 1: Preserve heading/body boundaries in correction application

**Files:** `src/foundry_mcp/core/research/workflows/deep_research/phases/claim_verification.py`
**Tests:** `tests/core/research/workflows/deep_research/test_claim_verification.py`

**Problem:** `_correct_single_claim()` sends a context window to the LLM that includes heading + body text. The LLM returns corrected text that concatenates heading and body onto a single line.

**Approach:** Add a post-correction validation step in `_correct_single_claim()` (after line 1070) that checks whether the `original_window` starts with or contains markdown headings (`^#{1,6}\s`), and if so, verifies the corrected text preserves the same heading structure. If a heading line in the original is followed by `\n` but the corresponding heading in the corrected text is followed by non-newline content, insert the missing `\n`.

Specifically:
- After `corrected_text = result.content.strip()` (line 1070), call a new helper `_repair_heading_boundaries(original_window, corrected_text)` that:
  1. Finds all heading lines (matching `^#{1,6}\s+.+$`) in the original window
  2. For each heading, checks if the corrected text has the same heading (possibly with modified text after it on the same line)
  3. Uses a regex to detect `^(#{1,6}\s+[^\n]+?)([A-Z])` patterns (heading immediately followed by uppercase letter with no newline) and inserts `\n\n` between them
  4. Returns the repaired text

**Edge cases:**
- Heading text itself may be legitimately modified by the correction — only enforce newline after heading, don't enforce identical heading content
- Multiple headings in a single window

### Fix 2: Strip inline LLM-generated source lists

**Files:** `src/foundry_mcp/core/research/workflows/deep_research/phases/_citation_postprocess.py`
**Tests:** `tests/core/research/workflows/deep_research/test_citation_postprocess.py`

**Problem:** `strip_llm_sources_section()` only matches `## Sources` / `## References` as markdown headings. The LLM sometimes generates an inline italicized source list like `*Sources: [2] [Title](url), [3] ...*` at the end of the report body.

**Approach:** Extend `strip_llm_sources_section()` to also strip inline source paragraphs:
1. After the existing heading-based stripping, add a second pass that matches patterns like:
   - `^\*?Sources\s*:\s*\[` (italicized or plain "Sources:" followed by citation references)
   - `^---\n+\*Sources:` (preceded by horizontal rule)
2. Remove the matched paragraph (to next `\n\n` or end of string)
3. Clean up any leftover `---` horizontal rule that preceded the stripped sources paragraph

**Test cases:**
- Report with only `## Sources` heading (existing behavior, no regression)
- Report with only inline `*Sources: [1] ...*` paragraph
- Report with both heading and inline sources
- Report with `---` rule before inline sources (strip both)
- Report with no sources at all

### Fix 3: Citation remapping for UNSUPPORTED claims

**Files:** `src/foundry_mcp/core/research/workflows/deep_research/phases/claim_verification.py`
**Tests:** `tests/core/research/workflows/deep_research/test_claim_verification.py`

**Problem:** 14/36 claims were marked UNSUPPORTED because the claim cited a source that doesn't contain the relevant evidence. The claims themselves are often factually correct — they just have wrong citations. Currently, only CONTRADICTED claims trigger corrections.

**Approach:** Add a citation-remapping step that runs after claim verification but before (or instead of) full correction:

1. Add a new function `remap_unsupported_citations()` in `claim_verification.py`:
   - For each `UNSUPPORTED` claim where `cited_sources` is non-empty:
     - Search `state.sources` for a source whose `raw_content` or `compressed_content` contains evidence supporting the claim text
     - If a better source is found, replace the citation number in the report text (simple `[old]` → `[new]` within the claim's quote_context region)
     - If no source supports the claim, optionally remove the citation or add a `[citation needed]` marker (configurable)
   - For each `PARTIALLY_SUPPORTED` claim:
     - Same logic, but only remap if the partial support comes from citation mismatch (not factual inaccuracy)

2. Wire into `workflow_execution.py` after claim verification:
   ```python
   if verification_result.claims_unsupported > 0:
       await remap_unsupported_citations(state, verification_result, execute_fn)
   ```

3. Track remapping stats in `ClaimVerificationResult`:
   - Add `citations_remapped: int = 0` field

**Complexity note:** This is the most involved fix. The source-matching step could use either:
- **LLM-based matching** (higher accuracy, higher cost): Send claim + candidate source contents to LLM, ask which source best supports the claim
- **Keyword/embedding matching** (cheaper, lower accuracy): Search source content for key terms from the claim

Recommend **LLM-based matching** for accuracy, with a token budget cap to avoid runaway costs. Batch claims per source to amortize prompt overhead.

### Fix 4: Strengthen synthesis prompt to reduce citation misattribution

**Files:** `src/foundry_mcp/core/research/workflows/deep_research/phases/synthesis.py`
**Tests:** `tests/core/research/workflows/test_synthesis_prompts.py`

**Problem:** The three citation-accuracy guardrails added in `9ebb084` are principles-level instructions. The LLM still misattributes when the source list is long and topics overlap.

**Approach:** Add structural reinforcement to the synthesis prompt:
1. In `_build_synthesis_tail()`, when appending the source reference list, add a per-source topic tag or brief content summary (1 sentence) next to each source citation number. This gives the LLM a quick-reference map to reduce guessing. Example:
   ```
   [1] NerdWallet — current transfer bonus percentages for all programs (March 2026)
   [2] LendingTree — Chase UR transfer partner list and ratios
   ```
2. Add an explicit instruction: "Before citing a source, verify the specific fact appears in that source's content above. If the fact came from your general knowledge rather than a specific source, do not cite any source."

**Token budget concern:** Adding per-source summaries increases prompt size. Use the existing `content_allocation` budget to limit summary length (truncate to ~15 words per source).

### Fix 5: Recalibrate fidelity_score to penalize contradictions more heavily

**Files:** `src/foundry_mcp/core/research/workflows/deep_research/models/deep_research.py`
**Tests:** `tests/core/research/workflows/deep_research/test_claim_verification.py`

**Problem:** The current `fidelity_score` formula treats CONTRADICTED at 0.0 and PARTIALLY_SUPPORTED at 0.5, but doesn't penalize contradictions — they simply contribute 0 to the numerator. A report with 10 supported + 2 contradicted scores 0.83, same as 10 supported + 2 unsupported. Contradictions (actively wrong) should score worse than unsupported (missing evidence).

**Approach:** Apply a negative weight to contradicted claims:
```python
@property
def fidelity_score(self) -> Optional[float]:
    if not self.claims_verified:
        return None
    score = (
        self.claims_supported
        + 0.5 * self.claims_partially_supported
        - 0.5 * self.claims_contradicted  # penalty
    ) / self.claims_verified
    return max(0.0, score)  # floor at 0
```

This makes contradicted claims actively drag down the score rather than being neutral.
